# PlaidBar 1.0 Roadmap

PlaidBar 1.0 should be the first release that can be recommended to a
privacy-conscious macOS user without a maintainer walking them through the
setup. The product promise is narrow by design: connect Plaid safely, keep data
local, make the most important personal-finance signals glanceable, and provide
clear recovery paths when something breaks.

This roadmap is the release contract between v0.5 and v1.0. It covers product,
design, frontend quality, local system behavior, security, distribution,
documentation, and open-source credibility.

## 1.0 Release Thesis

PlaidBar becomes a trustworthy open-source local finance utility, not only a
good demo.

1. The app must explain what it stores, where it stores it, and what it never
   sends to a hosted backend.
2. A new user must reach useful demo or sandbox data in minutes.
3. Production setup must be intentionally gated by Plaid approval, credentials,
   local server state, and explicit data-boundary copy.
4. Common breakage must be recoverable from the UI: server offline, wrong mode,
   missing credentials, expired Plaid item, empty account data, notification
   permission denied, and stale sync.
5. The repository must make a strong open-source case: clear architecture,
   screenshots, security model, contribution path, release process, and issue
   templates.

## SDLC Plan

### 1. Discover

Map the current user journey and contributor journey before adding breadth.

- Audit first launch from clean machine state.
- Audit demo mode without Plaid credentials.
- Audit sandbox with valid Plaid sandbox credentials.
- Audit production readiness without credentials, wrong credentials, and Plaid
  approval missing.
- Review every empty state and error state for a clear next action.
- Review README, PRD, security docs, release docs, and screenshots as a new
  contributor would.
- Maintain a living risk register in this file until 1.0.

Exit criteria:

- All critical user paths have a named owner surface and verification command.
- All known release blockers are listed under "1.0 Blockers".

### 2. Design

Keep the app menu-bar native: dense, restrained, glanceable, and action-oriented.

- Define final information architecture for:
  - setup/onboarding
  - dashboard filters
  - account drill-down
  - transactions and recurring charges
  - spending activity
  - credit utilization
  - status diagnostics
  - settings
- Standardize empty, degraded, loading, error, and permission-denied states.
- Keep settings as a utility surface, not a marketing page.
- Keep chart meaning available without color-only communication.
- Keep all destructive actions confirmation-gated.
- Make all primary controls keyboard reachable and VoiceOver-labeled.

Exit criteria:

- Every screen has a screenshot in `Assets/` generated from demo or sandbox
  data.
- Design tokens cover spacing, typography, semantic color, status badges, chart
  palettes, focus states, and destructive actions.
- No UI surface depends on invisible hover-only controls.

### 3. Build

Implement in small, reviewable slices. Product scope should stay focused.

- Prefer local, typed state and existing SwiftUI patterns.
- Keep Plaid network calls inside `PlaidBarServer`.
- Keep shared DTOs and pure utilities in `PlaidBarCore`.
- Keep `PlaidBar` as the native UI and local-client layer.
- Avoid cloud sync, analytics, hosted backends, and personal data export by
  default.
- Add tests when behavior crosses module boundaries or changes local-data
  semantics.

Exit criteria:

- Each milestone has a merged PR with passing CI.
- Every user-facing behavior change is reflected in PRD, README, or release
  notes.

### 4. Verify

Use a layered quality gate before each release candidate.

- `git diff --check`
- `bash -n Scripts/*.sh Scripts/plaidbar-run`
- `ruby -c Formula/plaidbar.rb`
- `swift build --target PlaidBar --skip-update --disable-keychain`
- `swift build -Xswiftc -strict-concurrency=complete -Xswiftc -warnings-as-errors --disable-keychain`
- `swift build -c release --disable-keychain`
- `swift test --skip-update --disable-keychain` when the local Swift toolchain
  supports the test framework
- `./Scripts/screenshots.sh`
- `./Scripts/smoke-sandbox.sh` with sandbox credentials or CI-safe smoke
  credentials
- Manual clean-profile launch
- Manual Settings > Accounts add/remove/reconnect check
- Manual notification permission check
- Secret/private-data scan over docs, screenshots metadata, source, tests, and
  scripts

Exit criteria:

- The 1.0 release candidate has a visible QA matrix and no unresolved critical
  blocker.

### 5. Release

Only tag 1.0 from a clean, verified `main`.

- Update `version.env`, `Info.plist`, runtime constants, tests, formula tag,
  release docs, and changelog.
- Verify Homebrew formula build.
- Produce GitHub release notes.
- Attach or link current screenshots.
- Mark known limitations honestly.
- Confirm support/security policy.

Exit criteria:

- `v1.0.0` tag exists.
- GitHub release is published.
- Homebrew install path works from source or formula.
- README quick start matches the tagged release.

### 6. Operate

Treat 1.0 as the start of a support loop.

- Triage issues weekly after launch.
- Label bugs by area: onboarding, server, Plaid, UI, settings, security,
  distribution, docs.
- Keep a short public changelog.
- Backport only critical fixes to patch releases.
- Avoid accepting broad product expansion until reliability is stable.

## Product Requirements

### First-Run and Onboarding

Requirements:

- Demo mode must work without Plaid credentials, network access, or a running
  server.
- Sandbox setup must show local server readiness before opening Plaid Link.
- Production setup must clearly say that real account data requires Plaid
  production approval.
- The app must explain local token/data storage before a Plaid link flow starts.
- After the user returns from Plaid Link, setup must verify linked item,
  accounts loaded, transaction sync attempted, and dashboard readiness before
  dismissing onboarding.
- Setup must offer recovery actions for:
  - server offline
  - wrong server mode
  - missing credentials
  - invalid Plaid Link URL
  - browser open failure
  - no linked items after returning
  - linked item present but accounts not loaded
  - accounts loaded but first transaction sync not complete

Acceptance criteria:

- A new user can launch `plaidbar --demo` and see dashboard data in under 30
  seconds.
- A developer with sandbox credentials can run `plaidbar-run --sandbox`, connect
  a sandbox item, complete the post-Link check, and land on the dashboard
  without editing source.
- No setup path appears clickable if it cannot complete.

### Dashboard

Requirements:

- Menu bar label modes remain configurable: net cash, total cash, credit
  utilization, recent spend, and icon-only.
- Dashboard shows:
  - net worth or selected menu bar metric
  - cash
  - debt
  - savings
  - credit exposure
  - sync health
  - 365-day spending activity
  - account filter chips
  - selected account details
- Account rows must show enough context to scan quickly: institution, account
  name, balance, status, utilization when relevant, and pending activity when
  relevant.
- Status filter must surface local server health and item health without
  burying it in Settings.
- Status filter must show a compact readiness panel with server mode, server
  state, credential readiness, linked item count, synced item count, last sync,
  and one clear recovery action for offline, stale, missing credentials,
  incomplete sync, login-required, and errored item states.

Acceptance criteria:

- Dashboard screenshots exist for Overview, Cash, Credit, Savings, Debt, and
  Status.
- Every footer icon button has a VoiceOver label.
- Empty and degraded dashboard states offer one clear next action.
- Status can recover common sync failures without forcing the user to find the
  matching Settings tab first.

### Transactions and Recurring Charges

Requirements:

- Transaction list supports date, category, and account filtering.
- Recurring detection remains explainable and conservative.
- Transaction detail sheet shows merchant, raw name, category, date, account,
  amount, pending/posted status, and confidence/source context where available.
- Filters should reset with the popover lifecycle unless persistence is
  deliberately introduced and documented.

Acceptance criteria:

- Filtered zero-result states include a clear filter-reset action.
- Recurring empty state explains the minimum history needed.
- Transaction detail is keyboard and VoiceOver reachable.

### Spending and Credit

Requirements:

- Spending activity must remain useful without overwhelming the menu bar.
- Spending heatmap must expose Spend and Net modes.
- Category colors must be documented and stable.
- Credit utilization should show thresholds, warnings, and high-risk cards.

Acceptance criteria:

- Charts have non-color text or accessibility labels for meaning.
- Large balances, negative balances, and zero-history states render without
  layout collapse.

### Settings

Requirements:

- Settings tabs remain General, Accounts, Notifications, and About unless a new
  tab is strongly justified.
- General includes launch at login, menu label preference, server endpoint, and
  local data controls.
- Accounts includes linked items, account counts, item health, reconnect, add
  account, and confirmation-gated removal.
- Notifications includes permission status, enablement, thresholds, and system
  settings guidance when macOS permission is denied.
- About includes version, build, repo link, privacy stance, and support links.

Acceptance criteria:

- Settings screenshots exist for Local Data, Accounts, and Notifications.
- Destructive reset/remove actions are confirmation-gated.
- Notification denied state does not leave controls looking enabled.

## Frontend and Design Improvements

### Visual System

- Finalize semantic colors for:
  - brand
  - positive
  - warning
  - negative
  - neutral
  - offline
  - syncing
  - stale
  - reconnect required
- Keep finance density high but readable.
- Keep popover width and content heights stable across loading states.
- Avoid oversized cards inside cards.
- Use clear status badges instead of long explanatory blocks.
- Keep icon buttons as icons with tooltips/accessibility labels.

### Interaction System

- Primary actions:
  - Add Account
  - Refresh
  - Settings
  - Reconnect
  - Check Again
  - Reveal Data Folder
  - Copy Path
- Destructive actions:
  - Remove account/item
  - Reset local data
  - Clear cached transactions, if added later
- Keyboard:
  - Cmd+R refresh
  - Cmd+N add account
  - Cmd+, settings
  - Escape dismiss sheets/popovers where appropriate
- Loading:
  - Server check
  - Sync in progress
  - Plaid Link opening
  - Data reset
- Empty states:
  - demo available
  - no linked item
  - server offline
  - no transactions
  - no recurring charges
  - filters returned zero
  - permission denied

### Accessibility

- Audit all icon-only buttons.
- Audit charts and badges for text alternatives.
- Preserve focus rings.
- Respect reduced motion for animations.
- Keep VoiceOver order aligned with visual order.
- Add accessibility labels to status chips and account rows.
- Avoid color-only warning semantics.

## System Architecture

### Current Shape

PlaidBar uses three major modules:

- `PlaidBar`: SwiftUI menu bar app and local UI state.
- `PlaidBarCore`: DTOs, pure utilities, formatting, sync reducers, local data
  models, and presentation helpers.
- `PlaidBarServer`: local Hummingbird companion server, Plaid API client,
  SQLite storage, token vault, and local auth.

### 1.0 Architecture Requirements

- Local server binds to `127.0.0.1` by default.
- App-server communication uses a locally generated auth token.
- Plaid credentials stay in server environment/config, not in the app binary.
- Plaid access tokens stay in local scoped stores under `~/.plaidbar/`.
- Sandbox and production stores remain separated.
- Legacy database migration remains explicit or safely inferred.
- UI surfaces must not need raw Plaid secrets.
- Server status endpoint must be safe to expose to the local app and should not
  leak secrets.
- Scripts must fail clearly when credentials or ports are wrong.

### System Reliability Work

- Add clearer duplicate-instance handling.
- Harden local server startup diagnostics.
- Keep endpoint configuration visible in Settings.
- Add structured error types for server-client failures.
- Add retry/backoff where sync failure semantics are known.
- Keep cached data visible during transient failures.
- Do not clear local data on Plaid or network errors.

## Security and Privacy Requirements

### Threat Model

Assets:

- Plaid `client_id` and `secret`
- Plaid `access_token`
- Item IDs and account IDs
- Account balances
- Transaction names, dates, amounts, categories, and merchants
- SQLite databases under `~/.plaidbar/`
- Local app-server auth token

Primary risks:

- Secret leakage through logs, screenshots, issues, or crash output.
- Sandbox/production data crossover.
- Local server exposed beyond localhost.
- Unauthorized local process calling the local server.
- Unsafe reset/removal behavior that destroys user expectations.
- Public examples accidentally containing real financial data.

Controls:

- Localhost binding.
- Local bearer token.
- Environment-scoped SQLite files.
- Explicit setup copy.
- Confirmation gates.
- Synthetic screenshots.
- Public vulnerability reporting guidance.
- No telemetry or hosted backend.

### 1.0 Security Checklist

- Confirm no logs print Plaid secrets, access tokens, public tokens, account
  numbers, or raw transaction exports.
- Confirm `/api/status` cannot expose secrets.
- Confirm unauthenticated API calls are rejected.
- Confirm local auth-token file permissions are reasonable.
- Confirm scoped database names prevent sandbox/production token crossover.
- Confirm reset flow explains what is local and what is not revoked at Plaid or
  the bank.
- Confirm screenshots use demo/sandbox data only.
- Confirm release archives do not include local `.plaidbar` data.

### Contribution Philosophy

PlaidBar should accept focused improvements that strengthen a local-first menu
bar finance utility. It should resist becoming:

- a cloud budgeting app
- an investment dashboard
- a multi-user finance platform
- a telemetry product
- a generic Plaid playground

Good first contribution areas:

- accessibility labels
- empty-state copy
- docs and troubleshooting
- screenshot coverage
- tests around pure utilities
- small UI polish using existing tokens

- Source build via SwiftPM.
- Homebrew formula for CLI/server/app executable install.
- Signed and notarized app bundle if the packaging path is completed before
  1.0.
- Sparkle update readiness only if the appcast and signing workflow are truly
  operational.

### Packaging Work

- Decide whether 1.0 ships as:
  - formula-only SwiftPM executable release, or
  - notarized `.app` archive plus formula/cask.
- If notarized app is included:
  - configure Developer ID signing
  - notarize archive
  - staple ticket
  - verify Gatekeeper on a clean machine
  - document Sparkle appcast requirements
- If formula-only:
  - be explicit that this is the first stable source-built release
  - defer cask/notarized bundle to 1.1

## Post-1.0 Scope

Strong candidates:

- CSV/JSON export with explicit redaction and confirmation.
- Category budget alerts.
- Better recurring charge explanations.
- Teller.io as an alternate provider.
- Desktop widget.
- Notarized cask release if formula-only shipped at 1.0.

Defer until product reliability is mature:

- Cloud sync.
- Multi-user support.
- iOS companion app.
- Investment tracking.
- AI/ML insights.
- Webhooks.

## Success Metrics

Release quality:

- CI green on release commit.
- No critical known setup blocker.
- Screenshot suite current.
- Security checklist complete.
- Homebrew install verified.

User value:

- Demo time to first data under 30 seconds.
- Sandbox time to first data under 3 minutes with credentials.
- Production setup explains all blockers before Plaid Link opens.
- Common failure states resolve without reading source code.

Open-source credibility:

- README can answer what it is, why it exists, how to install, how data is
  handled, how to contribute, and how to report security issues.
- A contributor can build, test, and run demo mode from docs alone.
- Public screenshots and fixtures contain no real financial data.
