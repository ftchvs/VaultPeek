# VaultPeek Long-Term Product Vision

VaultPeek (formerly PlaidBar) is a proprietary, local-first macOS menu bar
instrument for personal finance. It connects to Plaid, keeps user financial data on the user's machine,
and turns balances, spending activity, account health, and sync state into a
compact native surface.

This document is the long-term product brief for VaultPeek. It is intentionally
not a task tracker. It should be useful as the starting context for an agent,
maintainer, or internal collaborator deciding what work belongs in the product
and what should stay out.

## Current Product Truth

VaultPeek 1.0 shipped under the product's former name, PlaidBar. The current
stable shape is a privately-distributed, drag-install DMG for licensed macOS
users comfortable running a local-first Plaid utility.

The product is not a hosted finance app. It has no hosted backend, no telemetry,
no cloud sync, and no multi-user account model. Plaid credentials and linked
financial data are handled through local app and server boundaries, with token
storage and local databases documented in the repository.

The near-term design direction is inspired by RepoBar: a compact menu bar
utility that feels native, dense, glanceable, and status-rich. For VaultPeek, the
equivalent product surface is a finance instrument: heatmap first, account rows
second, recovery states always visible, local-only insight receipts explicit,
and settings as a clear local control plane.

Liquid Glass is the baseline surface treatment, not a progressive enhancement.
VaultPeek's minimum floor is macOS 26 (Tahoe), so the macOS 26 Liquid Glass APIs
(`Glass.regular`, `glassEffect`) are always available and used directly — no
material fallback path is required.

## North Star

VaultPeek should make a privacy-conscious Mac user feel in control of their
financial connection without turning into a budgeting suite.

The ideal user can glance at the menu bar, open a compact popover, understand
the state of their accounts, notice unusual spending or sync issues, and recover
from common local or Plaid failures without reading source code.

The ideal contributor can read the repository, understand the data boundary,
run demo mode, verify screenshots, and make a focused improvement without
needing private infrastructure.

## Product Principles

### Local First Is The Product

Local-first is not only an implementation detail. It is the core trust claim.
Every feature should preserve the idea that user financial data stays on the
user's machine unless the user explicitly connects to Plaid as part of normal
Plaid API use.

Good product behavior:

- Explain where local data lives.
- Explain what Plaid sees and what VaultPeek does not host.
- Keep sandbox and production data separate.
- Make destructive local-data actions explicit and confirmation-gated.
- Prefer cached local visibility during transient network or Plaid failures.

Bad product behavior:

- Hiding data storage details behind vague privacy copy.
- Adding analytics, telemetry, or remote state by default.
- Treating local reset, Plaid unlinking, and bank-side revocation as the same
  thing.
- Allowing convenience features to blur sandbox and production boundaries.

### Menu Bar First

> **Superseded by [ADR-001](strategy/macos26-migration/ADR-001-window-first-architecture.md) — window-first hybrid, accepted at Gate 0 (AND-578) 2026-06-19.** VaultPeek is now **window-first hybrid**: a primary window workspace is the main experience, with the menu-bar glance retained as a first-class *reduced read+route* surface. The guardrail below ("treat a large canvas / long workflow / persistent workspace with skepticism") now governs the **glance surface only** — session workflows (review, budgeting, planning, reconciliation) are first-class window destinations. The glance principles in this section still define the `MenuBarExtra`.

VaultPeek should remain a menu bar utility, not a full desktop finance app that
happens to have a menu bar icon.

The popover should answer a narrow set of questions quickly:

- What is my current financial posture?
- Did my accounts sync?
- Which account needs attention?
- Has recent spending changed?
- What is the one next action if something is broken?

Any feature that needs a large canvas, long workflow, or persistent workspace
should be treated with skepticism unless it strengthens the menu bar experience.

### Dense, Native, Readable

The visual target is a native macOS finance instrument:

- Compact rows over large decorative cards.
- A prominent 365-day spending activity heatmap.
- Clear account identity, balance, health, and utilization cues.
- Native material surfaces, separators, and restrained color.
- Icon actions with accessible labels and predictable keyboard behavior.
- Status badges that are short enough to scan and specific enough to act on.

VaultPeek should feel closer to a system utility than a marketing dashboard.
Polish should reduce cognitive load, not add decoration.

### Recovery Is A Primary Experience

Financial data tools fail in ordinary ways: local server offline, wrong mode,
missing credentials, expired item, stale sync, empty account list, no
transactions, notification permission denied, or a Plaid login-required state.

VaultPeek should treat those states as first-class product surfaces. A recovery
state is good when it names the problem, preserves safe context, and offers one
clear next action.

The user should not need to know which module failed before the UI can help.
Dashboard, setup, and settings should converge on the same recovery semantics.

### Privacy Trust Must Be Clear

The product's privacy posture is part of the value. VaultPeek asks users to
connect sensitive financial accounts, so its data-handling story must be
unusually clear — to licensed users and to internal collaborators.

The product should keep clearly documented:

- Architecture and data-flow documentation.
- Privacy and security claims that match the code.
- Troubleshooting paths for Plaid, local server, permissions, and setup.
- Synthetic screenshots and fixtures.
- Release notes and known limitations.
- Contributor guidance for focused improvements.

Trust erodes when docs overpromise, screenshots hide failure states, or release
notes imply distribution paths that are not actually supported.

## Target Experience

### The Glance

The menu bar label should be configurable and quiet. It can show net cash, total
cash, credit utilization, recent spend, or an icon-only mode. The label should
not become a ticker, notification feed, or high-noise alert system.

The glance succeeds when the user can decide whether to open the popover.

### The Popover

The main popover should be the product's core instrument.

The preferred order is:

1. Current posture and sync health.
2. Spending activity heatmap.
3. Compact filters.
4. Dense account rows.
5. Selected account detail.
6. Secondary summaries and footer actions.

This ordering keeps the most global signal visible while still making account
inspection fast. The popover should avoid large stacked cards that push account
state below the fold.

### Setup And Onboarding

First launch should make the user's data mode obvious:

- Demo mode should work without credentials.
- Sandbox mode should be reachable for developers with Plaid sandbox
  credentials.
- Production mode should clearly explain Plaid approval and local credential
  requirements before asking the user to proceed.

Onboarding should teach the local data boundary once, then use recovery states
instead of repeating long explanations.

### Settings

Settings are the control plane, not a secondary dashboard.

They should make local state understandable:

- General preferences.
- Accounts and item health.
- Notification permission and thresholds.
- Local data controls.
- Version, repository, privacy, and support links.

Destructive actions should be explicit. Permission-denied states should look
disabled or blocked rather than merely explained.

## Strategic Horizons

These horizons describe direction, not tickets. Hermes or any maintainer agent
should derive concrete tasks only after inspecting the current repository,
current PRs, CI state, and open issues.

### Horizon 1: Trust And Truth

VaultPeek must keep product claims, docs, screenshots, and behavior aligned.
This includes privacy copy, release notes, install paths, supported data modes,
and known limitations.

The important question is: would a new user or contributor understand what is
real today?

### Horizon 2: Native Finance Instrument

The dashboard should keep moving toward a compact, high-signal popover:
RepoBar-style density, a prominent heatmap, account rows that scan quickly, and
native macOS surfaces that remain legible in light, dark, and accessibility
modes.

The important question is: can the user understand account posture and sync
health in one glance?

### Horizon 3: Recovery Convergence

All common local, Plaid, account, transaction, and notification failures should
share clear recovery language and actions across setup, dashboard, and settings.

The important question is: does every broken state show one safe next action?

### Horizon 4: Distribution Confidence

Privately-distributed, ad-hoc-signed DMG distribution is the stable 1.0 path. A
notarized app bundle and a private Sparkle update channel should happen only
when signing, notarization, Gatekeeper verification, and release automation are
real.

The important question is: can the project support the distribution promise it
is making?

### Horizon 5: Focused Extensibility

VaultPeek can grow after the trust, recovery, and design loop is stable. Good
extensions should stay local-first, menu-bar-first, and explicit about data
boundaries.

Potential later extensions:

- Export with redaction and confirmation.
- More explainable recurring charge detection.
- Category or merchant-level alerts.
- A signed app distribution track.
- Carefully scoped alternate provider support.

The important question is: does the extension strengthen VaultPeek's core
instrument, or does it turn the product into something else?

## Product Boundaries

### Core Scope

VaultPeek should prioritize:

- Plaid account connection and local sync.
- Balances, cash, debt, savings, credit exposure, and recent spending.
- Spending activity over time.
- Account health and sync health.
- Local data controls.
- Notifications for simple thresholds and account attention.
- Source/developer local CLI access through the same authenticated localhost
  server.
- Local-only insight receipts and category-hint evidence that degrade clearly
  when no local model runtime is configured.
- Clear recovery actions.
- Product docs and release clarity.

### Allowed Later With Care

These ideas may fit if the core experience is stable:

- Redacted CSV or JSON export.
- Better recurring charge explanation.
- Category-level budget alerts.
- Expanded local AI summaries for the last 7 days, last month, and
  year-over-year financial activity, provided transaction data never leaves the
  Mac and deterministic/no-runtime receipts remain understandable.
- Local AI categorization suggestions for expenses and income, provided Plaid
  categories remain the auditable fallback and corrections are reversible.
- Signed app bundle distribution (Developer ID + notarization; Homebrew
  formula/cask distribution is discontinued).
- Desktop widget for a small subset of signals.
- Alternate Plaid-like provider support.

Each should require an explicit data-boundary review.

### Resist

These directions should remain out of scope unless the project deliberately
changes identity:

- Hosted backend.
- Cloud sync.
- Telemetry by default.
- Multi-user finance platform.
- iOS companion app.
- Investment portfolio dashboard.
- Cloud AI/ML over private transactions or any insight feature that sends raw
  financial data off-device.
- Generic Plaid developer playground.

## Architecture Vision

VaultPeek should keep its module boundaries easy to reason about:

- `PlaidBar` (SwiftPM target name kept for compatibility): native SwiftUI menu
  bar app, presentation, local client state, settings, and user recovery
  surfaces.
- `PlaidBarCore`: DTOs, pure utilities, formatting, sync reducers, local data
  models, and presentation helpers that can be tested without app runtime.
- `PlaidBarServer`: local Hummingbird companion server, Plaid API client,
  SQLite storage, token vault, and local auth.

The app should not need raw Plaid secrets. The server should bind locally by
default. App-server communication should use local authentication. Sandbox and
production stores should stay separated. Cached data should remain visible
during transient failures. Local data deletion should be deliberate and
understandable.

Architecture work is most valuable when it makes these boundaries clearer, not
when it adds abstraction for its own sake.

## Design System Vision

VaultPeek's design system should be small and durable:

- Semantic status colors for success, warning, negative, neutral, syncing,
  stale, offline, and reconnect-required states.
- Liquid Glass surface tokens built directly on the macOS 26 baseline APIs.
- Spacing and typography tuned for a menu bar popover.
- Account rows with stable height and predictable alignment.
- Charts and heatmaps with non-color accessibility alternatives.
- Icon-only controls with tooltips and accessibility labels.
- Confirmation patterns for destructive local-data actions.

The system should support future polish without letting visual experiments
change the product's trust posture.

## Development Operating Model

VaultPeek work — by the owner and authorized internal collaborators — should
focus on improvements that make the local finance instrument more trustworthy,
understandable, accessible, or resilient.

Good lower-risk work areas:

- Accessibility labels and VoiceOver order.
- Empty and degraded states.
- Screenshot coverage.
- Troubleshooting documentation.
- Tests for pure utilities and data reducers.
- Small UI polish using existing tokens.
- Release and packaging documentation.

High-risk work areas:

- Storage semantics.
- Plaid token handling.
- Production credential flow.
- Local server authentication.
- Destructive reset behavior.
- Distribution and signing claims.

High-risk changes should include explicit verification notes and docs updates.

## Hermes Agent Starting Point

When using this file as starting context, Hermes should treat it as a product
constitution rather than a backlog.

Before proposing or implementing work, Hermes should answer:

1. Does this strengthen the local-first menu bar finance instrument?
2. Does it preserve the data boundary and the product's privacy/trust claim?
3. Does it improve glanceability, recovery, native design, or internal
   collaborator clarity?
4. Is it supported by the current repository state, not only by this document?
5. Would it be better represented as a small PR, a doc correction, or a deferred
   idea?

Hermes should inspect current code, docs, branch state, PRs, CI, and issues
before generating tasks. If a task conflicts with this vision, Hermes should
either narrow it or mark it as out of scope.

Preferred Hermes output:

- A short product rationale.
- The affected user or contributor journey.
- The smallest useful implementation slice.
- Verification steps.
- Any privacy, security, accessibility, or distribution risk.
- Whether the work belongs now, later, or outside VaultPeek.

Hermes should avoid turning every horizon into tickets at once. The useful
agent behavior is to keep the long-term vision stable while choosing the next
small, defensible slice.

## Long-Term Success Signals

VaultPeek is succeeding when:

- A user can understand their account posture from the menu bar and popover.
- A broken state explains itself and offers a clear recovery action.
- Demo and sandbox flows let collaborators verify behavior without real data.
- Privacy and security claims match implementation.
- Screenshots show real product states, including degraded states.
- The repository remains approachable to authorized internal collaborators.
- Distribution claims are conservative and true.
- New features make the core instrument better rather than broader for its own
  sake.
