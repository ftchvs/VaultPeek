# Support & Operations Runbook

> **Audience: operators.** This is the OPERATOR-facing playbook for handling
> VaultPeek (formerly PlaidBar) support requests. It is **not** a user-facing
> self-help guide. For self-help, point users to
> [`docs/troubleshooting.md`](troubleshooting.md) and do not duplicate its steps
> here — this runbook tells the operator how to *triage and respond*, then links
> the user to the right self-help section.

VaultPeek is a local-first macOS menu bar dashboard for Plaid financial data.
Two processes run on the user's Mac: the SwiftUI app and a companion
[Hummingbird](https://github.com/hummingbird-project/hummingbird) server bound to
`127.0.0.1:8484`. Data lives in local SQLite plus the macOS Keychain. There is
**no cloud backend, no telemetry, and no hosted log store** (see
[`docs/privacy.md`](privacy.md)). That means the operator **cannot look anything
up server-side**. Every diagnosis comes from what the user reports, what the
in-app surfaces say, and the redacted readiness metadata at `/api/status`.

## The one rule that governs everything below

VaultPeek's whole value proposition is "your financial data stays on your Mac."
Support must never undercut that. **Never ask a user — and never accept if
offered — any of the following:**

- Plaid `client_secret`, `client_id`, `access_token`, or `public_token`
- The local app/server bearer token (`auth-token`)
- Plaid `item_id` / `account_id` / `transaction_id` / `institution_id`
- Real balances, real transactions, or real merchant history
- Raw SQLite database files, transaction exports, or Keychain contents
- Full server logs without confirming they are redacted first

These are the exact categories the product is built to keep private
([`SECURITY.md`](../SECURITY.md) "Data Handling Boundaries",
[`docs/privacy.md`](privacy.md) "Screenshots and Issues"). If a user pastes any
of them, treat it as an exposure: tell them to rotate/revoke as appropriate and
do not retain the message. Suspected vulnerabilities or private-data exposure go
through private vulnerability reporting per [`SECURITY.md`](../SECURITY.md),
**never** a public issue.

The app is designed to make safe support possible: error detail shown to users
is already sanitized by `UserFacingError.sanitizedDetail`
(`Sources/PlaidBarCore/Utilities/UserFacingError.swift`), which redacts bearer
tokens, Plaid tokens, IDs, and balances out of any message before it is
displayed. So a screenshot of an in-app error banner is generally safe to
request — a raw server log line is not, unless the user confirms it is redacted.

## 1. Bank-connection failure taxonomy

VaultPeek surfaces connection health through a small, fixed set of states. The
app models exactly three item connection states
(`ItemConnectionStatus` in `Sources/PlaidBarCore/Models/LinkResponse.swift`):
`connected`, `login_required`, and `error`. The recovery copy a user sees is
generated by `ItemRecoveryTarget` and `ReconnectRecoveryMessage`
(`Sources/PlaidBarCore/Utilities/`). Use those concepts as the shared vocabulary
with the user — ask "what does the Status view say?", not "what's your item ID?".

`ItemRecoveryTarget.item(from:)` prioritizes any item in `.error` first, then the
first `.loginRequired` item, and produces a reconnect action titled
**"Reconnect <Institution>"** (or "Reconnect Item" when the institution name is
unknown). That is the single recovery affordance the user should be guided to in
**Status** or **Settings > Accounts**.

| Symptom (what the user sees) | What is actually wrong | Local or Plaid-side | Operator response |
|---|---|---|---|
| **Plaid Link closed / nothing got linked** ("I closed the browser tab" / "I changed my mind") | The Hosted Link flow never completed, so no item was created. No error state — the account list just stays empty. | Neither — the flow was abandoned client-side. | Reassure: nothing is broken, no item exists yet. Have them reopen setup / **Settings > Accounts** and start the link again. If the browser never opened at all, that is the `ReconnectRecoveryMessage.browserOpenFailed` path → "set a default browser, then retry" (see row below). |
| **"Missing or expired link session state"** error page after returning from the bank | The one-time Hosted Link `state` was already consumed or expired before the callback returned (validated in `LinkRoutes.handleCallback`; one-time state managed by `PendingLinkSessionStore`). | Local server-side (expected security behavior, not a bug). | Tell the user this is the expected single-use protection. Have them start a **fresh** link from the app — do not try to reuse the old browser tab/URL. |
| **"Reconnect required" / a bank shows the reconnect action** | Plaid marked the item `ITEM_LOGIN_REQUIRED`. The server maps this to `.loginRequired` (`AccountRoutes.itemStatus(for:)`, `TransactionRoutes.itemStatus(for:)`). Recovery copy: *"Plaid requires a fresh <Institution> login before account rows can be recovered."* (`ItemRecoveryTarget.recoveryDetail`). | Plaid-side: the bank requires re-authentication (MFA, password change, new terms). VaultPeek cannot fix it locally. | Walk the user through the self-help: [`docs/troubleshooting.md` → "Reconnect Is Required"]. The action is **Status / Settings > Accounts → Reconnect <Institution>**, which opens Plaid Hosted Link update mode (`POST /api/link/update/:itemId`), then **Refresh** after returning. |
| **A bank shows an item error (not login_required)** | Plaid returned an item error other than login-required. Server maps any non-`ITEM_LOGIN_REQUIRED` failure to `.error`. Recovery copy: *"Plaid reported an item error for <Institution>. Reconnect it, then refresh balances."* | Plaid-side (item/institution problem). | Same path as reconnect: **Reconnect <Institution>**, then **Refresh**. If reconnect repeatedly fails for one specific institution, this is likely a Plaid/institution issue VaultPeek cannot resolve — see "Institution unsupported / Plaid-side" below. |
| **"VaultPeek could not open Plaid Link in your browser"** | `ReconnectRecoveryMessage.browserOpenFailed` — the app could not hand the Link URL to a browser. | Local (no default browser set / browser launch failed). | Have the user set a default browser in macOS, then retry **Settings > Accounts → Reconnect <Institution>**. |
| **"VaultPeek could not prepare a safe reconnect link"** | `ReconnectRecoveryMessage.invalidUpdateLinkURL` — the update-mode link URL failed validation before opening. | Local (link preparation guard). | Have the user **Refresh status** first, then retry the reconnect action. If it persists, collect a safe bug report (Section 4). |
| **"VaultPeek could not create a reconnect link. <detail>"** | `ReconnectRecoveryMessage.createFailed` — `POST /api/link/update/:itemId` failed. The `<detail>` shown is already sanitized (≤80 chars, tokens/IDs/balances stripped). | Could be local (server) or Plaid-side; the sanitized detail hints which. | Read the sanitized detail back with the user. Refresh status, retry reconnect. If it still fails, escalate with the safe report. |
| **Transient failures: "Plaid API error", refreshes intermittently fail, recover on retry** | The Plaid client retries on HTTP **429 (rate limit)** and **5xx** (`PlaidClient.isRetryableHTTPStatus` retries `429` and `500–599`). Persistent 429s mean Plaid is rate-limiting this caller. | Plaid-side throttling / transient outage. | Advise the user to **wait and retry** rather than mashing Refresh (repeated manual balance refreshes hit a per-call billed endpoint and worsen throttling). If it's a sustained outage, it is Plaid-side and outside VaultPeek's control. |
| **Accounts linked but transactions empty** | Plaid may not have returned history yet, the item needs another sync, the account truly has no history, filters are hiding rows, or app/server point at different data dirs/environments. | Mixed. | Do not treat as data loss. Route to [`docs/troubleshooting.md` → "Accounts Are Linked But Transactions Are Empty"]: Refresh, clear filters, check Status, confirm sandbox vs production. |

**Institution unsupported / "my bank isn't available" / Plaid-side limits.**
VaultPeek cannot add institution coverage, change Plaid product behavior, or
grant Plaid production access. Per [`SUPPORT.md`](../SUPPORT.md) "Plaid-Specific
Support": VaultPeek can make local server state and setup readiness visible, but
institution coverage and production approval are **Plaid's** domain. Operator
response: confirm the user is in the right mode (sandbox vs production), confirm
their Plaid app supports the institution, and direct institution-coverage
questions to Plaid documentation/support.

**Sandbox vs production confusion ("I linked accounts but production shows
none").** This is expected, not data loss. Sandbox and production use strictly
separate local stores (`plaidbar-sandbox.sqlite` vs
`plaidbar-production.sqlite`) and separate caches; each mode starts empty until
linked in that mode. Route to
[`docs/troubleshooting.md` → "Production Mode Shows No Linked Accounts After
Sandbox Testing"].

## 2. Billing states & refund / cancellation handling

> **TBD — pending billing implementation (AND-393).** Stripe billing is **gated
> and not implemented in the product today.** Do **not** describe, promise, or
> act on any billing behavior as if it exists. This section is an operator
> *response framework* for the questions you will get, not a description of
> shipped behavior.

What is true today, grounded in code and docs:

- **There is no billing, no charging, and no subscription enforcement.**
  Every connection today is **bring-your-own (BYO) Plaid keys** or demo data
  ([`docs/privacy.md`](privacy.md) "Managed Bank Linking (Planned — Not Yet
  Available)"). BYO and demo are intended to stay free.
- **The in-app plan picker is a preview only.** Per
  [`docs/privacy.md`](privacy.md): "The app's plan picker is a **preview** of
  proposed tiers; selecting a plan changes nothing, charges nothing, and grants
  no access." Treat any "managed plan" copy in the app as forward-looking.
- **The pricing tiers, refund policy, and managed bridge are proposed, not
  decided.** Tier prices, the 30-day refund default, and managed linking are
  **open decisions awaiting approval** — do not quote them to users as policy.

Operator response framework:

| User asks | Operator response today |
|---|---|
| "Did you charge me?" / "Will I be charged?" | No. VaultPeek does not charge anyone today; there is no billing system. Selecting a plan in the app charges nothing and grants nothing — it is a preview. **TBD — pending billing implementation (AND-393).** |
| "How do I cancel my subscription?" | There is no subscription to cancel today. If they used BYO keys, their data is already entirely on their Mac; they can stop using the app and optionally reset local data (Section 3). Paid cancellation flow is **TBD — pending billing implementation (AND-393).** |
| "I want a refund." | There is nothing billed to refund today. A refund policy is **proposed but not decided** and is **TBD — pending billing implementation (AND-393).** Do not promise a window or terms. |
| "What does my plan include / what are the prices?" | Proposed prices are **estimates**, not published prices. Do not quote them as committed. The only thing live today is free BYO-keys and demo. **TBD — pending billing implementation (AND-393).** |
| "Is my data on your servers now that I'm on a paid plan?" | No managed/hosted mode exists today; nothing is hosted. (See Section 5 privacy snippet, and the important caveat for the *future* managed mode in `docs/privacy.md`.) |

Do not improvise billing behavior. If a user has a billing question this
framework does not cover, escalate it as a product question tagged AND-393 —
do not invent an answer.

### Managed bank-link support boundary (planned)

> **TBD — pending managed-link implementation (AND-394).** Support-assisted
> managed linking is not available today. Before it can be sold, the consent,
> audit, and escalation rules must be implemented and approved.

The planned boundary is strict: a helper may guide a user through VaultPeek
screens and record a safe audit event, but must never ask for, receive, type, or

balances, transactions, local databases, logs, or screenshots containing real
financial data. If a case cannot be diagnosed without one of those inputs, it
escalates; it does not become a reason to collect the input.

## 3. Local data path, reset/recovery, and migration troubleshooting

All user data is local. Default location
([`docs/privacy.md`](privacy.md), [`SECURITY.md`](../SECURITY.md),
`README.md`):

```text
~/.vaultpeek/
```

Override with the `PLAIDBAR_DATA_DIR` environment variable. The same directory
holds, by default:

- `auth-token` — local app↔server bearer token (**never ask the user to send
  this**)
- `server.conf` — local config (may contain `PLAID_CLIENT_ID`/`PLAID_SECRET`;
  **never ask for its contents**)
- `plaidbar-sandbox.sqlite` / `plaidbar-production.sqlite` — environment-scoped
  item stores (filenames intentionally keep the `plaidbar-` prefix)
- account/transaction caches, pending link-session state, `server.log`

Plaid access-token **bytes** live in macOS Keychain under the service
`PlaidBar.PlaidAccessToken` when Security framework support is available; SQLite
holds only `keychain:<item_id>` references ([`SECURITY.md`](../SECURITY.md)).

### Server restart / "Server is offline"

The bundled server is auto-started by the app and can restart independently.
Route the user to [`docs/troubleshooting.md` → "App Says Server Is Offline"].
Key operator checks to confirm with the user:

- Health endpoint responds: `http://127.0.0.1:8484/health` (default port).
- App and server use the **same** port; custom ports need
  `PLAIDBAR_SERVER_PORT` exported before launching the app.
- The user can read **readiness metadata** safely: an authenticated
  `GET /api/status` returns only version, environment, item/synced counts, last
  sync, credential availability, and storage path (`ServerStatus` in
  `Sources/PlaidBarCore/Models/ServerStatus.swift`) — **no** secrets, IDs,
  balances, or transactions. This is the one server-state surface you can safely
  ask a user to describe.

### Duplicate / legacy instance conflicts

A still-installed legacy **PlaidBar.app** shares the same bundle identifier and
default port `8484` with **VaultPeek.app** and will conflict. Route to
[`docs/troubleshooting.md` → "App Says Server Is Offline"]: quit and delete the
legacy app, quit duplicate VaultPeek app/server processes.

### Migration: `~/.plaidbar/` → `~/.vaultpeek/`

On startup, default installs copy **missing** files forward from `~/.plaidbar/`
into `~/.vaultpeek/` and **never overwrite** newer `~/.vaultpeek/` files; the
legacy directory is left in place as rollback evidence. Keychain entries are
**not** renamed, so `keychain:<item_id>` references keep working. If a user
reports "my data didn't migrate," route to
[`docs/troubleshooting.md` → "Default Storage Did Not Migrate"] and "API Calls
Return Unauthorized" (auth-token mismatch). Temporary rollback is
`PLAIDBAR_DATA_DIR=~/.plaidbar` before launching **both** app and server.

> **Important:** do not have the user paste any migrated file. Migration
> troubleshooting is about *which file is missing where*, confirmable by the
> user locally — never by sending you the file.

### Local data reset (and its limits)

A local reset clears VaultPeek-owned database files, account/transaction caches,
pending Link sessions, and stored Plaid access-token entries; it leaves
`server.conf`, auth, and preferences untouched
([`docs/privacy.md`](privacy.md) "Local Reset Boundary"). After a reset, a reset
marker prevents old data from being copied back from `~/.plaidbar/`.

**Critical thing to tell the user:** local reset does **not** revoke bank access
in Plaid Dashboard or at the bank. Route to
[`docs/troubleshooting.md` → "Local Data Reset Did Not Revoke Bank Access"]: for
complete revocation, the user must also review Plaid Dashboard and the bank's
connected-app permissions.

## 4. Collecting a useful bug report without requesting financial secrets

The goal is enough signal to reproduce, with zero private financial data. The
app already sanitizes user-visible error text
(`UserFacingError.sanitizedDetail`), so screenshots of in-app banners are
usually safe; raw logs and database contents are not.

**DO ask for:**

- macOS version and VaultPeek version (version is shown in `/api/status` and
  the About surface — safe metadata).
- **Mode**: demo, sandbox, or production.
- The exact in-app error text or a screenshot of the **error banner** (already
  sanitized).
- The connection state words the user sees in **Status**: `connected`,
  `login_required` (shown as "reconnect required"), or `error`.
- What `/api/status` reports, described in words: credentials configured y/n,
  environment, item count, synced count, last sync time, sync ready y/n. (This
  endpoint is contractually secret-free.)
- Steps to reproduce, using **demo or sandbox** data.
- For server/log issues: confirm the server is running and `/health` responds.

**DO NOT ask for (and refuse if offered):**

- Plaid `client_secret` / `client_id` / `access_token` / `public_token`
- The `auth-token` file or its contents
- Plaid `item_id` / `account_id` / `transaction_id` / `institution_id`
- Real balances, real transactions, or real merchant names
- SQLite database files (`plaidbar-*.sqlite`), sidecars, or transaction exports
- Keychain contents (`PlaidBar.PlaidAccessToken`)
- `server.conf` contents or environment dumps (these can carry `PLAID_SECRET`,
  which can also leak via shell history — [`SECURITY.md`](../SECURITY.md))
- **Unredacted** `server.log`. If a log is genuinely needed, instruct the user to
  reproduce in **sandbox or demo** mode and confirm the snippet contains no
  tokens, IDs, balances, or account numbers before sending.

This mirrors the "Do not include" lists in [`SUPPORT.md`](../SUPPORT.md),
[`docs/privacy.md`](privacy.md), and [`SECURITY.md`](../SECURITY.md). When in
doubt, prefer demo/sandbox reproductions and redacted snippets.

## 5. Standard reply snippets

Ready-to-send, operator-safe replies. Adjust tone as needed; do not add any
secret-collection step.

### Privacy — "Is my data uploaded anywhere?"

> Short answer: no. VaultPeek is local-first — there's no VaultPeek cloud
> backend, no analytics, and no telemetry. Your balances and transaction history
> live only on your Mac (a local database, with bank tokens in your macOS
> Keychain). The only thing that leaves your Mac is the call your *own* local
> server makes directly to Plaid to fetch your data when you're in sandbox or
> production mode — in demo mode nothing is sent to Plaid at all. We can't see
> your data because there's nowhere on our side for it to go.

*(A future optional "managed" mode is on the roadmap and would change part of
this story; it does not exist today. If asked specifically about a "managed
plan," see `docs/privacy.md` "Managed Bank Linking (Planned — Not Yet
Available)" before answering.)*

### Bank won't connect / reconnect required

> Thanks — this is usually fixable in a minute and doesn't mean anything was
> lost. Open VaultPeek, go to **Status** (or **Settings > Accounts**), and look
> for the bank showing a reconnect prompt. Click **Reconnect <your bank>**,
> complete the Plaid login in your browser, then click **Refresh** when you're
> back in VaultPeek. If the browser doesn't open, set a default browser in macOS
> and try the reconnect again. Full steps are in our troubleshooting guide under
> "Reconnect Is Required." Please don't send us any account numbers or
> tokens — a screenshot of the error banner is all we'd ever need.

### Data reset

> Resetting clears VaultPeek's local data on your Mac (databases, cached
> accounts/transactions, pending link sessions, and stored bank tokens). It does
> **not** disconnect you at the bank or in Plaid — so if your goal is to fully
> revoke access, also remove VaultPeek from your bank's connected-apps settings
> and from the Plaid Dashboard. Your config and preferences are left in place.
> See "Local Data Reset Did Not Revoke Bank Access" in our troubleshooting
> guide. There's nothing you need to send us to do this — it all happens locally
> on your machine.

### First launch on macOS — "macOS won't let me open the app" (Gatekeeper)

> That's expected for this build. VaultPeek's DMG is ad-hoc-signed and not yet
> notarized, so on first launch macOS Gatekeeper blocks a normal double-click.
> The one-time workaround: **right-click (or Control-click) VaultPeek.app in
> Applications and choose Open**, then confirm. After that first time it opens
> normally. (Notarized distribution is planned but not shipped yet, so please
> keep using right-click → Open for now.)

*(Grounding: [`docs/release.md`](release.md), [`docs/distribution.md`](distribution.md)
— ad-hoc-signed DMG, right-click → Open required, notarization deferred. Do not
tell users it is notarized.)*

### "How do I cancel?" (billing TBD)

> Right now there's nothing to cancel — VaultPeek doesn't charge anyone today.
> If you connected a bank, you did it with your own Plaid keys (bring-your-own),
> which is free, and all your data is already on your Mac. You can simply stop
> using the app, and if you'd like, reset your local data (it stays on your
> machine). Paid plans and cancellation/refund handling aren't built yet, so
> there's no charge to reverse. **(Billing is not implemented — TBD pending
> billing work, AND-393.)**

## Escalation & boundaries

- **Security / private-data exposure:** never a public issue. Use GitHub private
  vulnerability reporting or follow [`SECURITY.md`](../SECURITY.md).
- **Plaid coverage / production approval / Plaid API behavior:** VaultPeek can't
  change these; direct to Plaid ([`SUPPORT.md`](../SUPPORT.md)).
- **Billing questions beyond Section 2:** escalate as a product question tagged
  AND-393; do not improvise.
- **Managed-link consent, audit, account removal, or helper-boundary questions:**
  managed support is not live. Escalate as a product question tagged AND-394.
- **Anything requiring a secret to diagnose:** stop. If you cannot diagnose it
  from sanitized in-app text, `/api/status` words, and demo/sandbox repros, it
  escalates to engineering with a safe repro — it does not escalate by
  collecting the secret.

If you can't resolve a request without one of the forbidden inputs in Section 1,
that is a signal to escalate, not to ask for the secret.
